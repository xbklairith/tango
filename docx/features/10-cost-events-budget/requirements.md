# Requirements: Cost Events & Budget Enforcement

**Created:** 2026-03-15
**Status:** Draft

## Overview

Track cost events generated by agent operations (LLM calls, tool usage, API calls) and enforce budget limits per agent and per squad. When monthly spend reaches 80% of the budget limit, a soft alert inbox item is created. At 100%, a hard stop auto-pauses the agent. This enables financial governance of AI agent workforces with full attribution to agent, issue, project, and goal.

### Requirement ID Format

- Use sequential IDs: `REQ-001`, `REQ-002`, etc.
- Keep numbering continuous across all requirement categories.

---

## Data Schema

### `cost_events` Table

| Column           | Type        | Nullable | Description                                                      |
|------------------|-------------|----------|------------------------------------------------------------------|
| id               | UUID        | NO       | Primary key, auto-generated                                      |
| squad_id         | UUID FK     | NO       | Squad scope; references `squads(id) ON DELETE CASCADE`          |
| agent_id         | UUID FK     | NO       | Agent that incurred the cost; references `agents(id)`           |
| heartbeat_run_id | UUID FK     | YES      | Which run generated this cost; references `heartbeat_runs(id)`  |
| issue_id         | UUID FK     | YES      | Cost allocation to issue                                         |
| project_id       | UUID FK     | YES      | Cost allocation to project                                       |
| goal_id          | UUID FK     | YES      | Cost allocation to goal                                          |
| event_type       | VARCHAR(50) | NO       | Type of cost-generating operation (see Event Types below)       |
| model            | VARCHAR(100)| YES      | Model identifier (e.g., `claude-3-5-sonnet-20241022`)           |
| provider         | VARCHAR(50) | YES      | Provider name (e.g., `anthropic`, `openai`)                     |
| input_tokens     | INT         | YES      | Number of input/prompt tokens consumed                          |
| output_tokens    | INT         | YES      | Number of output/completion tokens generated                    |
| cost_cents       | BIGINT      | NO       | Normalized cost in USD cents (must be >= 0)                     |
| billing_code     | VARCHAR(100)| YES      | Custom cost allocation tag provided by agent or user            |
| usage_json       | JSONB       | YES      | Raw token counts, model metadata, provider-specific fields      |
| created_at       | TIMESTAMPTZ | NO       | Immutable event timestamp (default: `now()`)                    |

**Indexes:**
- `(squad_id, created_at DESC)` — for squad spend aggregation filtered by month
- `(agent_id, created_at DESC)` — for per-agent spend aggregation filtered by month
- `(squad_id, agent_id, created_at DESC)` — composite for agent-within-squad queries

**Constraints:**
- No `UPDATE` or `DELETE` routes exist for this table; it is append-only by design.
- `cost_cents >= 0` check constraint.

### Event Types (`event_type`)

| Value        | Description                                |
|--------------|--------------------------------------------|
| `llm_call`   | LLM inference (input + output tokens)      |
| `tool_use`   | External tool or API invocation            |
| `embedding`  | Embedding model call                       |
| `other`      | Miscellaneous cost-generating operation    |

## Functional Requirements

### Event-Driven Requirements (WHEN…THEN)

- **[REQ-001]** WHEN an agent runtime completes a cost-generating operation (LLM call, tool use, embedding) THEN the system SHALL record a `cost_event` row containing: `squad_id`, `agent_id`, `event_type`, `cost_cents`, `model`, `provider`, `input_tokens`, `output_tokens`, `usage_json`, and `created_at`.

- **[REQ-002]** WHEN a `cost_event` is recorded THEN the system SHALL immediately compute the agent's running monthly spend total by summing `cost_cents` for that `agent_id` where `created_at >= start of current UTC calendar month`.

- **[REQ-003]** WHEN a `cost_event` is recorded AND the agent has a `budget_monthly_cents` set AND the agent's new running spend total is >= 80% but < 100% of that budget THEN the system SHALL create an inbox alert item of category `alert` and type `budget_warning` for the squad, if no unresolved `budget_warning` alert already exists for that agent in the current month.

- **[REQ-004]** WHEN a `cost_event` is recorded AND the agent has a `budget_monthly_cents` set AND the agent's new running spend total is >= 100% of that budget THEN the system SHALL auto-pause the agent by transitioning its status from `running` or `idle` to `paused` and SHALL record an activity log entry with action `agent.budget_paused`.

- **[REQ-005]** WHEN a `cost_event` is recorded AND the squad has a `budget_monthly_cents` set AND the squad's new running spend total (sum across all agents in the squad) is >= 80% but < 100% of that budget THEN the system SHALL create a squad-level inbox alert of type `budget_warning`, if no unresolved squad-level `budget_warning` alert already exists for the current month.

- **[REQ-006]** WHEN a `cost_event` is recorded AND the squad has a `budget_monthly_cents` set AND the squad's new running spend total is >= 100% of that budget THEN the system SHALL auto-pause all `running` or `idle` agents in the squad and SHALL record an activity log entry per agent with action `agent.budget_paused`.

- **[REQ-007]** WHEN the budget settings for an agent are updated via `PATCH /api/agents/:id` THEN the system SHALL re-evaluate the current month's spend total against the new budget and apply the appropriate threshold action (no action, soft alert, or auto-pause) atomically within the same transaction.

- **[REQ-008]** WHEN the budget settings for a squad are updated via `PATCH /api/squads/:id/budgets` THEN the system SHALL re-evaluate the current month's squad spend total against the new budget and apply the appropriate threshold action atomically within the same transaction.

- **[REQ-009]** WHEN `GET /api/squads/:id/costs/summary` is called THEN the system SHALL return: total spend this month, budget limit, percentage utilised, threshold status (`ok`, `warning`, `exceeded`), and the billing period start/end dates.

- **[REQ-010]** WHEN `GET /api/squads/:id/costs/by-agent` is called THEN the system SHALL return a list of per-agent cost summaries including: `agent_id`, `agent_name`, `spend_cents` for the current month, `budget_monthly_cents`, and `threshold_status`.

- **[REQ-011]** WHEN `POST /api/squads/:id/cost-events` is called with a valid payload THEN the system SHALL persist the event, check thresholds, and return `201 Created` with the created event body; the enforcement check SHALL occur within the same database transaction as the insert.

- **[REQ-012]** WHEN `GET /api/agent/me/budget` is called with a valid Run Token JWT THEN the system SHALL return the calling agent's spend this month, its `budget_monthly_cents`, percentage utilised, and current threshold status, scoped strictly to the agent identified by the token.

---

### State-Driven Requirements (WHILE…the system SHALL)

- **[REQ-013]** WHILE an agent's status is `running`, the system SHALL accept and record all cost events submitted for that agent without rejecting them on the basis of budget state (enforcement happens after recording, not before).

- **[REQ-014]** WHILE an agent's status is `paused` due to budget enforcement (auto-paused), the system SHALL reject any attempt to transition that agent to `running` or `idle` unless the agent's or the squad's `budget_monthly_cents` has been increased (or removed) so that current month spend falls below 100% of the new limit.

- **[REQ-015]** WHILE the current calendar month is active, the system SHALL compute all spend totals using `SUM(cost_cents) WHERE created_at >= start_of_month_utc AND created_at < end_of_month_utc` — no cached running totals that could become stale.

---

### Ubiquitous Requirements (The system SHALL always)

- **[REQ-016]** The system SHALL store `cost_cents` as a non-negative integer in USD cents (e.g., $1.00 = 100).

- **[REQ-017]** The system SHALL scope all cost event reads and spend aggregations to `squad_id`, preventing cross-squad data access.

- **[REQ-018]** The system SHALL treat cost events as immutable: no `UPDATE` or `DELETE` operations are permitted on the `cost_events` table.

- **[REQ-019]** The system SHALL enforce budget thresholds atomically using a database transaction that includes both the cost event `INSERT` and any resulting agent status `UPDATE`, preventing race conditions that could allow spend to significantly overshoot the hard limit.

- **[REQ-020]** The system SHALL include `agent_id` on every cost event, enabling full attribution of spend to the agent that incurred it.

- **[REQ-021]** The system SHALL support optional cost allocation fields (`issue_id`, `project_id`, `goal_id`, `billing_code`) on every cost event so that spend can be attributed to work items.

- **[REQ-022]** The system SHALL define a "month" as a UTC calendar month (midnight UTC on the 1st through the last second of the last day) for all spend computations.

---

### Conditional Requirements (IF…THEN)

- **[REQ-023]** IF an agent has no `budget_monthly_cents` set (NULL) THEN the system SHALL skip all budget enforcement checks for that agent, allowing unlimited spend.

- **[REQ-024]** IF a squad has no `budget_monthly_cents` set (NULL) THEN the system SHALL skip squad-level budget enforcement, regardless of individual agent budgets.

- **[REQ-025]** IF a budget is increased (or removed) so that current month spend now falls below 100% of the new limit THEN the system SHALL allow a previously budget-paused agent to be manually resumed (transitioned to `active`) by a squad member with sufficient permissions.

- **[REQ-026]** IF a budget is increased (or removed) so that current month spend now falls below 80% of the new limit THEN the system SHALL resolve (dismiss) any outstanding `budget_warning` inbox alert for that agent or squad.

- **[REQ-027]** IF an agent was auto-paused due to budget enforcement THEN the system SHALL require a squad owner to explicitly resume the agent after budget has been increased; the system SHALL NOT auto-resume the agent.

- **[REQ-028]** IF a cost event payload includes `input_tokens` and `output_tokens` THEN the system SHALL store them in the `cost_events` table; however, their presence is optional and does not block event recording.

- **[REQ-029]** IF cost event recording is requested for an agent that has status `terminated` THEN the system SHALL reject the request with `409 Conflict` and error code `AGENT_TERMINATED`.

- **[REQ-030]** IF cost event recording is requested for an agent that does not belong to the squad identified in the URL THEN the system SHALL reject the request with `403 Forbidden` and error code `FORBIDDEN`.

## Non-Functional Requirements

### Performance

- **[REQ-031]** The system SHALL resolve spend aggregation queries (monthly sum per agent, monthly sum per squad) in under 200ms at p95 for squads with up to 1,000 agents and 100,000 cost events per month, using indexed queries.

- **[REQ-032]** The system SHALL complete the cost event insert + budget enforcement check + optional agent status update within a single database transaction that completes in under 500ms at p95.

### Security

- **[REQ-033]** The system SHALL require a valid Run Token JWT to call `POST /api/squads/:id/cost-events` and `GET /api/agent/me/budget`; the token must be scoped to the same squad as the cost event.

- **[REQ-034]** The system SHALL require squad membership (any role) to read cost summaries via `GET /api/squads/:id/costs/*`.

- **[REQ-035]** The system SHALL require squad owner role to update budget settings via `PATCH /api/squads/:id/budgets` and `PATCH /api/agents/:id` (budget field).

### Reliability

- **[REQ-036]** The system SHALL NOT lose a cost event in the case of a failed budget enforcement check; if the threshold evaluation fails after the insert succeeds, the insert SHALL be rolled back and the caller SHALL receive a `500 Internal Server Error` (they may retry).

- **[REQ-037]** The system SHALL be idempotent on cost event submission if the caller provides a client-generated `id` (UUID); a duplicate `id` SHALL return `200 OK` with the existing event rather than creating a duplicate.

---

## Constraints

- Cost events are immutable (append-only); no UPDATE or DELETE routes exist.
- Budget enforcement must use atomic operations (CAS / database transaction) to prevent overshoot.
- Both per-agent and per-squad budget limits must be supported and independently enforced.
- Budget amounts are in USD cents (integer); no floating-point currency values in the database.
- The `PATCH /api/squads/:id/budgets` endpoint already exists and must remain backward-compatible.
- Budget periods are always UTC calendar months in v1; custom periods are out of scope.
- Auto-pause writes must respect the existing agent status machine in `internal/domain/agent.go`.

---

## Acceptance Criteria

- [ ] `cost_events` table exists with all required columns and indexes.
- [ ] `POST /api/squads/:id/cost-events` records an event and returns 201.
- [ ] Monthly spend is correctly computed by summing `cost_cents` for the current UTC month.
- [ ] An agent with spend at 80% of `budget_monthly_cents` triggers a `budget_warning` inbox alert.
- [ ] An agent with spend at 100% of `budget_monthly_cents` is auto-paused and activity log entry is created.
- [ ] A squad with spend at 80% of `budget_monthly_cents` triggers a squad-level `budget_warning` inbox alert.
- [ ] A squad with spend at 100% of `budget_monthly_cents` auto-pauses all running/idle agents.
- [ ] `GET /api/squads/:id/costs/summary` returns correct totals, percentages, and threshold status.
- [ ] `GET /api/squads/:id/costs/by-agent` returns per-agent cost breakdown with threshold status.
- [ ] `GET /api/agent/me/budget` returns agent-scoped spend and budget, gated by Run Token.
- [ ] Budget increase re-evaluates thresholds and resolves outstanding alerts if spend drops below 80%.
- [ ] Budget-paused agent cannot be resumed without budget increase; manual resume by owner is required.
- [ ] Cost events for a terminated agent are rejected with 409.
- [ ] Cross-squad cost event access returns 403.
- [ ] No cost event is ever updated or deleted via the API.

---

## Out of Scope

- Billing integration with external payment providers (Stripe, etc.)
- Cost forecasting and prediction
- Multi-currency support
- Custom budget periods (rolling 30-day windows; v1 is calendar month only)
- Automated budget top-ups or self-service budget increases by agents

## Dependencies

- Agent status machine (`internal/domain/agent.go`) — for auto-pause transitions (`running`/`idle` → `paused`)
- Squad budget endpoint (`PATCH /api/squads/:id/budgets`) — already exists in `internal/server/handlers/squad_handler.go`
- Database migrations (goose) — new migration required for `cost_events` table
- Inbox/alert system — for `budget_warning` alert creation (feature dependency)
- Activity log — for `agent.budget_paused` log entries (feature dependency)

---

## Risks & Assumptions

**Assumptions:**
- Cost events arrive near-real-time from the agent runtime adapter (not batched by the adapter).
- All monetary amounts are denominated in USD cents; no multi-currency requirement in v1.
- A "budget month" is the UTC calendar month; no per-squad timezone configuration.
- The inbox/alert table exists or will be implemented before or alongside this feature.

**Risks:**
- High-frequency cost events (many agents making rapid LLM calls) may create hot-row contention; the mitigation is to avoid running totals stored as columns and rely entirely on aggregation queries with good indexing.
- If the inbox/alert system is not yet implemented, the soft-alert requirement (REQ-003, REQ-005) cannot be fulfilled; a fallback is to log the warning to the activity log instead.
- An agent crash between `cost_event` INSERT and the budget enforcement check could leave the event recorded without enforcement; the transaction wrapping (REQ-019) mitigates this — if enforcement fails, the insert is rolled back.
